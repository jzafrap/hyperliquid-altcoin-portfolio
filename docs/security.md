# Security & Trust Model

**Bottom line:** the app never stores your wallet's private key. It signs orders
with a delegated, **trade-only** key that lives only in memory and cannot move your
funds.

## Two keys, two roles

| Key | Who holds it | What it can do |
|-----|--------------|----------------|
| Your wallet key | Your wallet (Rabby/MetaMask) — never exposed to the app | Signs **one** agent approval. |
| Agent key | Generated by the app, kept **in memory only** | Places/cancels orders. **Cannot withdraw or transfer funds.** |

This is Hyperliquid's **agent (API) wallet** model. You approve an agent once; it
trades on your behalf within permissions that exclude moving funds out.

## Lifecycle

1. **Connect** wallet.
2. **Approve trading agent** — your wallet signs one `approveAgent` action. The app
   generates the agent keypair at this moment.
3. **Trade** — every buy/sell is signed locally by the agent key, no popups.
4. **Refresh / disconnect / switch wallet** — the agent key is discarded. You
   re-approve next session.

The UI states all of this at approval time, including that a refresh requires
re-approval.

## Where the key lives

- The agent private key is a **module-scoped variable** in `lib/agent.ts`.
- It is **never** written to `localStorage`, `sessionStorage`, cookies, or
  IndexedDB; **never** logged; **never** placed in serializable React state.
- Only a key-free snapshot (approved flag + public addresses) is exposed to the UI,
  via `useSyncExternalStore`.

## The trust boundary

Authorization is enforced where signing happens, not in the UI:

```ts
getAgentExchangeClient(masterAddress)   // throws unless an approved agent
                                        // is bound to this exact master
```

Because the current wallet address is passed at execution time and verified against
the agent's bound master, there is **no window** where a stale agent (approved for
wallet A) could sign an order attributed to a newly-connected wallet B — even before
UI cleanup runs.

Additional hardening from adversarial review:

- Re-validate after the async approval signature; discard the result if the wallet
  changed mid-approval.
- De-duplicate concurrent approvals; short-circuit if already approved.
- The agent client is narrowed to a **trade-only** surface (`order`, `cancel`,
  `cancelByCloid`, and `updateLeverage` for perp leverage selection) — no
  withdrawal/transfer methods are reachable through it.

## Money-safety in trading

See [Trading Model → Money-safety guarantees](./trading-model.md#money-safety-guarantees-enforced-in-executebuy--executesell):
never overspend, never throw after a fill, surface unsaved fills loudly, re-check
balance before executing.

## Known v1 limitations

Documented and accepted for the serverless v1 (see `instructions.md` §8.1):

| Limitation | Impact | Planned fix |
|------------|--------|-------------|
| Cross-tab lot writes not atomic | Two tabs trading the same wallet at the same instant can lose one update's **local** bookkeeping (on-exchange fills stay correct). | `navigator.locks` + re-read/merge. |
| Corrupt storage reads as empty | `loadLots` returns `[]` on a parse error, looking like "no positions". | Distinguish and warn. |
| No fee headroom in the funds guard | A near-exact-balance buy could be rejected by the exchange. | Add a fee buffer. |
